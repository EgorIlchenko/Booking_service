# Booking Service

Backend-сервис записи на встречи. Бронь создаётся через REST API в статусе
`pending`, ставится в очередь и асинхронно подтверждается фоновым воркером
(с имитацией сбоя внешнего сервиса ~15%), после чего логируется отправка
уведомления.

**Стек:** FastAPI, TaskIQ + Redis, PostgreSQL, SQLAlchemy 2.0 (async), Alembic,
Dishka (DI), structlog. Менеджер пакетов — uv.

## Запуск

Требуется Docker с Docker Compose.

```bash
cp .env.example .env
docker compose up --build
```

Поднимаются PostgreSQL, Redis, API и воркер. Миграции применяются автоматически
отдельным шагом `migrate` до старта приложения.

Короткие команды: `make up` — запуск стека, `make down` — остановка и удаление
контейнеров.

После старта:

- API — http://localhost:8000
- Swagger UI — http://localhost:8000/docs

### Пример запроса

```bash
curl -X POST http://localhost:8000/api/v1/bookings \
  -H "Content-Type: application/json" \
  -d '{"name": "Иван", "datetime": "2030-01-01T12:00:00+00:00", "service_type": "haircut"}'

# статус брони (воркер переведёт её в confirmed или failed)
curl http://localhost:8000/api/v1/bookings/<id>
```

### Эндпоинты

| Метод  | Путь                    | Описание                                                   |
|--------|-------------------------|------------------------------------------------------------|
| POST   | `/api/v1/bookings`      | Создаёт бронь в статусе `pending` и ставит задачу на подтверждение |
| GET    | `/api/v1/bookings/{id}` | Возвращает бронь; `404`, если не найдена                   |
| GET    | `/api/v1/bookings`      | Список с фильтром `?status=` и пагинацией `?limit=&offset=` |
| DELETE | `/api/v1/bookings/{id}` | Отмена брони; допустима только из `pending`, иначе `409`    |
| GET    | `/health`               | Проверка доступности                                       |

`POST /bookings` ограничен по частоте запросов (лимит задаётся в `.env`).

## Тесты

Запускаются без Docker: база — SQLite в памяти, Redis — fakeredis, очередь и
внешний сервис подменяются тестовыми реализациями.

```bash
uv sync
cp .env.example .env
uv run pytest
```

Эквивалент — `make test`. Статический анализ — `make lint` (ruff + mypy). Те же
проверки выполняются в CI на каждый pull request.

Структура тестов:

- `tests/unit` — бизнес-логика на подменах (создание, отмена, обработка, идемпотентность, валидация);
- `tests/integration` — репозиторий на реальном SQLAlchemy и rate limiter на fakeredis;
- `tests/e2e` — HTTP-эндпоинты через ASGI-клиент с подменённым DI-контейнером.

## Технические решения

### Фреймворк

FastAPI выбран как async-фреймворк с валидацией на Pydantic и автогенерацией
OpenAPI/Swagger из коробки — это минимизирует объём шаблонного кода в веб-слое.

Фоновая обработка построена на TaskIQ. Он изначально асинхронный и выполняется
в том же event loop, что и приложение, поэтому сервис и репозиторий
переиспользуются без изменений. Альтернатива — Celery — синхронная и потребовала
бы вызова `asyncio.run` на каждую задачу и отдельного пула соединений к БД.

### Структура

Приложение разделено на слои, зависимости направлены внутрь:

```
HTTP (FastAPI)  →  Service (бизнес-логика)  →  Repository (доступ к данным)
```

- веб-слой тонкий: валидация запроса, вызов сервиса, маппинг результата и доменных исключений в HTTP;
- сервис не зависит ни от FastAPI, ни от SQLAlchemy и обращается к хранилищу через интерфейс (`Protocol`), что упрощает тестирование;
- доменные исключения (`BookingNotFound`, `BookingNotCancellable`) выбрасываются в сервисе и преобразуются в коды `404`/`409` обработчиками веб-слоя.

Внедрение зависимостей реализовано через Dishka, а не штатный `Depends` FastAPI:
`Depends` действует только в пределах HTTP-запроса, тогда как сервис и
репозиторий нужны и в воркере, где запроса нет. Dishka предоставляет единый
контейнер для обоих входов (API и воркер), за счёт чего конфигурация
зависимостей не дублируется и не привязана к веб-слою.

Внешние взаимодействия скрыты за интерфейсами: очередь подтверждения и внешний
сервис (с вероятностью сбоя ~15%) описаны как `Protocol`-порты. В рабочем режиме
за ними стоят TaskIQ и генератор случайных значений, в тестах — подмены; сервис
от конкретных реализаций не зависит.

### Идемпотентность

Задача подтверждения может быть доставлена повторно (ретрай, передоставка из
очереди), что не должно приводить к рассогласованию данных. Алгоритм обработки:

1. бронь загружается запросом `SELECT ... FOR UPDATE` — строка блокируется на время транзакции и недоступна параллельному воркеру;
2. если бронь отсутствует (отменена) или уже не в статусе `pending` (обработана ранее) — обработка завершается без изменений;
3. из статуса `pending` бронь переводится в `confirmed` или `failed`, после чего транзакция фиксируется.

Повторный вызов с тем же `booking_id` не создаёт дубликат и не перезаписывает
уже установленный статус.

Бронь фиксируется в БД до постановки задачи в очередь: при обратном порядке
воркер мог бы получить задачу раньше, чем зафиксируется транзакция API, и не
найти запись.

Ретраи в воркере выполняются только при технических сбоях (например,
недоступность БД) с экспоненциальным backoff. Имитированный сбой внешнего
сервиса — это штатный исход со статусом `failed`, а не исключение, и ретрай не
вызывает.

## Логирование

structlog в формате JSON, вывод в stdout и в файлы с ротацией (`logs/`). Каждому
HTTP-запросу присваивается `request_id`, который попадает во все его логи и
позволяет проследить обработку. Уведомления об успешном подтверждении
дополнительно пишутся в отдельный файл `logs/notifications.log`.
